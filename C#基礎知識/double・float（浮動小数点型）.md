# double / float（浮動小数点型）

`0.1` や `0.2` のような値は、コンピュータ内部では正確に表現できないことがあります。
金額計算などで「正確な値」が必要なときは `decimal` 型を使いましょう（計算速度が `double` 型より劣るなどのデメリットはあります）。

---

## なぜ 0.1 が正確に表現できないのか？

- コンピュータは2進数で計算する
- `0.1` を2進数にすると無限小数になり切り捨てられる
- わずかな誤差が生まれる

## なぜ decimal は正確なのか？

10進数のまま内部で管理するから。

## なぜ decimal は遅いのか？

CPU が直接計算できず、ソフトウェアで処理するから。

## StringBuilder

文字列をループ文の中で繰り返し連結するような場合には、処理速度の観点から `+` 記号での連結ではなく、`StringBuilder` の使用が推奨されます。

### string の + 連結（非推奨）

- 毎回新しい文字列をメモリに作る
- ループが増えるほど遅くなる
- 5回ループで5個の文字列がメモリに作られる
- 古い文字列はゴミとしてメモリに残る

→ ループ回数が増えるほどメモリの無駄遣い・処理が遅くなります。

### StringBuilder（推奨）

- 内部バッファに追記するだけ
- 最後に `ToString()` で1回だけ文字列化
- ループでの連結に最適

## var の使いどき

実務では `var` を使うことが多いですが、右辺の型が読み手に伝わりにくい場合は明示的に型を書く。

```csharp
var score = CalculateScore();      // ❌ メソッド名から戻り値がわかりにくい
double score = CalculateScore();   // ✅ 型が明示されている
```

## ループの使い分け

| ループ | 使いどき |
| --- | --- |
| `foreach` | リストや配列などのコレクションを1つずつ取り出して処理する |
| `for` | 繰り返す回数を明確に指定したいとき・インデックスを使いたいとき |
| `while` | 「何回繰り返すか」が決まっていない処理 |

- `continue`：その回の処理をスキップして次に進む
- `break`：ループ自体を終了する

---

## コレクションの注意点

- リストなどのコレクションが `null` だと、`foreach` 実行時に null 参照エラーが発生する → null を返さず空リストを返す
- `foreach` ループ中にコレクションを変更（要素の追加や削除）するとエラーが発生する → LINQ の `Where` で新しいリストを作る

## LINQ・ラムダ式

- **LINQ**：リストなどのコレクションを簡単に操作するための機能
- **ラムダ式**：「関数（処理のかたまり）」をその場で簡潔に書くための記法

| メソッド | 用途 |
| --- | --- |
| `Where` | 絞り込み |
| `Select` | 変換 |
| `OrderBy` | 並び替え |
| `First` | 1件取得 |
| `Any` | 存在確認 |

最後に `.ToList()` をつけると `List<T>` に変換される。

### 即時評価と遅延評価

| 用途 | 方法 |
| --- | --- |
| 結果を何度も使う | `ToList()` で即時評価 |
| 一度だけ使う | 遅延評価のまま `foreach` |

## プロパティ・メソッド

- **プロパティ**：クラスのデータ（値）に対して外部からの読み取りや書き込みを行うための仕組み
- **静的メソッド**：`new` したインスタンスからではなく、`クラス名.静的メソッド名` で呼び出す
- **インスタンスメソッド**：そのオブジェクトに紐づくデータを使った処理に使う
- **静的メソッド**：特定のデータに依存しない共通処理に使う

```
「外に公開する → init」
「クラス内部だけ → readonly」
```

---

## 例外処理の注意

- 例外はエラーの内容を記録したり、追加処理を行ってから `throw;` を使う（握りつぶさない）
- 非同期メソッドを呼び出すときは必ず `await` を付ける
- 非同期メソッドを `async void` で定義しない（エラーを `catch` できず、プログラムが予期せず終了することがある）
