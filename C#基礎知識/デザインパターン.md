# デザインパターン

よく使われる設計の「型」。問題に名前が付いていることで、チームで共通認識を持ちやすくなる。

---

## 1. Repository パターン

データアクセスのロジックをビジネスロジックから分離する。

```csharp
// インターフェース（契約）
public interface IOrderRepository
{
    Task<Order?> GetByIdAsync(int id);
    Task<List<Order>> GetAllAsync();
    Task AddAsync(Order order);
    Task DeleteAsync(int id);
}

// 実装（SQL Server 版）
public class SqlOrderRepository : IOrderRepository
{
    private readonly AppDbContext _db;

    public SqlOrderRepository(AppDbContext db)
    {
        _db = db;
    }

    public async Task<Order?> GetByIdAsync(int id)
        => await _db.Orders.FindAsync(id);

    public async Task<List<Order>> GetAllAsync()
        => await _db.Orders.ToListAsync();

    public async Task AddAsync(Order order)
    {
        _db.Orders.Add(order);
        await _db.SaveChangesAsync();
    }

    public async Task DeleteAsync(int id)
    {
        var order = await _db.Orders.FindAsync(id);
        if (order != null)
        {
            _db.Orders.Remove(order);
            await _db.SaveChangesAsync();
        }
    }
}
```

**メリット：**
- テスト時にモック（偽の実装）に差し替えられる
- DBの種類を変えても Service 層のコードは変えなくて済む

## 2. Singleton パターン

インスタンスをアプリ全体で1つだけ作る。設定・ログ・キャッシュなどに使う。

```csharp
public class AppSettings
{
    // 唯一のインスタンス
    private static readonly AppSettings _instance = new AppSettings();

    // 外部からのインスタンス生成を禁止
    private AppSettings() { }

    public static AppSettings Instance => _instance;

    public string ConnectionString { get; set; } = string.Empty;
}

// 使い方
AppSettings.Instance.ConnectionString = "Server=...";
```

> C# の DI コンテナ（`services.AddSingleton<T>()`）を使う場合は、自分で Singleton を実装する必要はない。

## 3. Factory パターン

オブジェクトの生成を専用のクラスに任せる。

```csharp
public interface INotification
{
    void Send(string message);
}

public class EmailNotification : INotification
{
    public void Send(string message) => Console.WriteLine($"メール送信: {message}");
}

public class SlackNotification : INotification
{
    public void Send(string message) => Console.WriteLine($"Slack送信: {message}");
}

// Factory：種類に応じて適切なオブジェクトを作る
public static class NotificationFactory
{
    public static INotification Create(string type) => type switch
    {
        "email" => new EmailNotification(),
        "slack" => new SlackNotification(),
        _ => throw new ArgumentException($"未知の通知タイプ: {type}")
    };
}

// 使い方
var notification = NotificationFactory.Create("slack");
notification.Send("デプロイ完了！");
```

## 4. Strategy パターン

アルゴリズム（処理方法）を差し替え可能にする。

```csharp
// 「並び替え方法」というインターフェース
public interface ISortStrategy
{
    List<int> Sort(List<int> data);
}

public class AscendingSort : ISortStrategy
{
    public List<int> Sort(List<int> data) => data.OrderBy(x => x).ToList();
}

public class DescendingSort : ISortStrategy
{
    public List<int> Sort(List<int> data) => data.OrderByDescending(x => x).ToList();
}

// Strategy を使う側
public class Sorter
{
    private ISortStrategy _strategy;

    public Sorter(ISortStrategy strategy)
    {
        _strategy = strategy;
    }

    // 実行中に戦略を切り替えることもできる
    public void SetStrategy(ISortStrategy strategy) => _strategy = strategy;

    public List<int> Execute(List<int> data) => _strategy.Sort(data);
}

// 使い方
var sorter = new Sorter(new AscendingSort());
var result = sorter.Execute(new List<int> { 3, 1, 4, 1, 5 });
```

## 5. Observer パターン

状態変化を他のオブジェクトに自動通知する。C# の `event` / `INotifyPropertyChanged` がこのパターン。

```csharp
// イベントを発行する側（Subject）
public class StockPrice
{
    private decimal _price;

    // 価格変更イベント
    public event Action<decimal>? PriceChanged;

    public decimal Price
    {
        get => _price;
        set
        {
            _price = value;
            PriceChanged?.Invoke(_price);  // 購読者全員に通知
        }
    }
}

// 購読する側（Observer）
var stock = new StockPrice();
stock.PriceChanged += price => Console.WriteLine($"価格が {price} 円に変わった");
stock.PriceChanged += price => { if (price < 1000) Console.WriteLine("安値アラート！"); };

stock.Price = 950;  // → 2つのハンドラが自動実行される
```

## 6. Decorator パターン

既存クラスを変更せずに機能を追加する。

```csharp
public interface ILogger
{
    void Log(string message);
}

public class ConsoleLogger : ILogger
{
    public void Log(string message) => Console.WriteLine(message);
}

// タイムスタンプを付加するデコレータ
public class TimestampLogger : ILogger
{
    private readonly ILogger _inner;

    public TimestampLogger(ILogger inner)
    {
        _inner = inner;
    }

    public void Log(string message)
        => _inner.Log($"[{DateTime.Now:HH:mm:ss}] {message}");
}

// 使い方
ILogger logger = new TimestampLogger(new ConsoleLogger());
logger.Log("処理開始");  // → [10:30:00] 処理開始
```

---

## 7. パターンの選び方

| 課題 | 使うパターン |
| --- | --- |
| データアクセスをテストしやすくしたい | Repository |
| 1つのインスタンスを共有したい | Singleton |
| 生成ロジックを一箇所にまとめたい | Factory |
| 処理のアルゴリズムを差し替えたい | Strategy |
| 状態変化を他クラスに通知したい | Observer |
| 既存クラスに機能を追加したい | Decorator |
