# セッション管理の不備

---

## 1. セッション管理の不備とは

ウェブアプリケーションはHTTPのステートレスな性質を補うために**セッションID**でユーザーを識別する。このセッションIDの管理が不十分な場合、攻撃者がセッションIDを入手・悪用してなりすましを行う。

### 主な攻撃手法

| 攻撃手法 | 内容 |
| --- | --- |
| **セッションIDの推測** | 連番・短い文字列など予測しやすいIDを総当たりで試す |
| **セッションIDの盗用** | XSS・通信の盗聴・URLからの漏えいでIDを取得する |
| **セッションIDの固定化（Session Fixation）** | 攻撃者が事前にセッションIDをセットし、被害者がログインした後にそのIDを使ってなりすます |

---

## 2. 発生しうる脅威

攻撃が成功すると、攻撃者は**利用者になりすまし**、その利用者に許可されている操作を不正に行う。

### ログイン後のサービスの悪用

- 不正な送金・振込
- 利用者が意図しない商品購入
- 利用者が意図しない退会処理

### 情報の改ざん・新規登録

- 各種設定の不正な変更（管理者画面・パスワード等）
- 掲示板への不適切な書き込み

### 非公開情報の閲覧

- 非公開の個人情報を不正閲覧
- ウェブメールを不正閲覧
- 会員専用コンテンツを不正閲覧

---

## 3. 注意が必要なウェブサイトの特徴

ログイン機能を持つウェブサイト全般に注意が必要。特にログイン後に重要な処理を行うサイトは被害が大きくなる。

| カテゴリ | 例 |
| --- | --- |
| 金銭処理が発生するサイト | ネットバンキング・ネット証券・ショッピング・オークション |
| 非公開情報を扱うサイト | 転職サイト・コミュニティサイト・ウェブメール |
| その他ログイン機能を持つサイト | 管理者画面・会員専用サイト・日記サイト |

---

## 4. 対策1：セッションIDを推測が困難なものにする

セッションIDは十分な長さ・ランダム性を持たせ、攻撃者が総当たりで当てられないようにする。

```csharp
// ❌ NG：連番や短い固定値はすぐ推測される
var sessionId = userId.ToString();  // "1001", "1002" ... 総当たりで突破される

// ✅ OK：暗号論的に安全な乱数を使う（ASP.NET Core はセッション管理を自動で行う）
// ASP.NET Core のセッションミドルウェアは内部で安全なIDを生成する
builder.Services.AddSession(options =>
{
    options.IdleTimeout    = TimeSpan.FromMinutes(20);
    options.Cookie.HttpOnly = true;
    options.Cookie.IsEssential = true;
});
```

---

## 5. 対策2：セッションIDをURLパラメータに格納しない

URLにセッションIDを含めると、Refererヘッダー・ブラウザ履歴・ログから漏えいする。

```
// ❌ NG：URLにセッションIDが含まれる
https://example.com/mypage?sessionid=abc123

// ✅ OK：Cookie に格納する（URLに出現しない）
Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Strict
```

---

## 6. 対策3：Cookie に適切な属性を設定する

```csharp
// ASP.NET Core での Cookie 属性設定
builder.Services.AddSession(options =>
{
    options.Cookie.HttpOnly  = true;   // JavaScript からアクセス不可（XSS対策）
    options.Cookie.SecurePolicy = CookieSecurePolicy.Always;  // HTTPS のみで送信
    options.Cookie.SameSite = SameSiteMode.Strict;  // クロスサイトリクエストで送信しない
    options.IdleTimeout     = TimeSpan.FromMinutes(20);
});
```

| 属性 | 効果 |
| --- | --- |
| `HttpOnly` | JavaScript から Cookie を読み取れなくする（XSS による盗用を防ぐ） |
| `Secure` | HTTPS 通信時のみ Cookie を送信する（盗聴を防ぐ） |
| `SameSite=Strict` | 別サイトからのリクエストでは Cookie を送信しない（CSRF を防ぐ） |

---

## 7. 対策4：ログイン成功後に新しいセッションを開始する（Session Fixation 対策）

ログイン前のセッションIDをログイン後も継続して使用すると、Session Fixation 攻撃が成立する。

```
攻撃の流れ（Session Fixation）：
1. 攻撃者が自分のセッションIDを取得：sessionid=XYZ
2. 被害者に sessionid=XYZ をセットしたURLを踏ませる
3. 被害者がそのIDでログイン成功
4. 攻撃者も sessionid=XYZ でアクセス → なりすまし成功
```

```csharp
// ✅ OK：ログイン成功後に古いセッションを破棄し、新しいセッションを発行する
public async Task<IActionResult> Login(LoginModel model)
{
    var user = await _authService.ValidateAsync(model.Username, model.Password);
    if (user == null)
        return Unauthorized();

    // 古いセッションを破棄
    HttpContext.Session.Clear();

    // 新しいセッションにユーザー情報を格納
    await HttpContext.Session.LoadAsync();
    HttpContext.Session.SetString("UserId", user.Id.ToString());

    return RedirectToAction("Index", "Home");
}
```

## 8. 対策5：ページ遷移ごとに秘密情報を確認する

ログイン成功後にセッションIDとは別の**トークン**を発行し、各ページで確認することで、セッションIDが盗まれても被害を抑える（二重チェック）。

```csharp
// ログイン時にトークンを発行
var token = Convert.ToBase64String(RandomNumberGenerator.GetBytes(32));
HttpContext.Session.SetString("CsrfToken", token);
ViewData["CsrfToken"] = token;  // フォームに hidden フィールドで埋め込む

// リクエスト受信時に照合
var sessionToken = HttpContext.Session.GetString("CsrfToken");
var formToken    = Request.Form["csrf_token"];

if (sessionToken != formToken)
{
    return Forbid();  // トークン不一致 → 拒否
}
```

## 9. 対策6：セッションIDを固定値にしない／有効期限を設定する

```csharp
builder.Services.AddSession(options =>
{
    // 一定時間操作がなければセッションを無効化
    options.IdleTimeout = TimeSpan.FromMinutes(20);

    // ブラウザを閉じたら Cookie を削除（セッション Cookie）
    options.Cookie.MaxAge = null;  // 明示的な有効期限を設定しない = セッション Cookie
});
```

| 設定 | 推奨 |
| --- | --- |
| セッションの有効期限 | 20〜30分のアイドルタイムアウトを設ける |
| ログアウト時 | セッションを完全に破棄する |
| 長期ログイン | `Remember me` はセッションとは別の仕組みで実装する |

---

## 10. まとめ：セッション管理の不備 対策チェックリスト

| 対策 | 内容 |
| --- | --- |
| IDを推測困難にする | 暗号論的に安全な乱数でセッションIDを生成する |
| URLに含めない | セッションIDは Cookie にのみ格納する |
| Cookie 属性を設定 | `HttpOnly` / `Secure` / `SameSite` を必ず付ける |
| ログイン後にIDを再発行 | Session Fixation 対策として古いセッションを破棄する |
| トークンで二重チェック | セッションIDとは別のトークンをページ遷移ごとに照合する |
| 有効期限を設定 | アイドルタイムアウトを設け、ログアウト時にセッションを破棄する |
