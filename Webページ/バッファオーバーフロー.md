# バッファオーバーフロー

---

## 1. バッファオーバーフローとは

プログラムが確保したメモリ領域（バッファ）を超えたデータを書き込むことで、**隣接するメモリ領域を上書きし、プログラムの動作を乗っ取る**攻撃。

```
正常時：
バッファ [A][A][A][A][A][A][A][A] | 戻りアドレス [正常]

攻撃時（バッファ超過）：
[A][A][A][A][A][A][A][A][攻撃コード...→] | 戻りアドレス [書き換え]
                                           ↑ 攻撃者が指定したアドレスに実行が移る
```

---

## 2. 発生しうる脅威

- **任意コードの実行**：攻撃者が用意したコードをサーバー上で実行される
- **サーバーの乗っ取り**：管理者権限の取得・バックドアの設置
- **サービス不能（DoS）**：プログラムのクラッシュによるサービス停止
- **情報漏えい**：メモリ内の機密情報（パスワード・セッション等）の読み取り

---

## 3. 注意が必要なウェブサイトの特徴

- C・C++ などメモリ管理を手動で行う言語で実装されたウェブアプリケーション
- 外部からの入力を固定長バッファにコピーする処理が含まれるプログラム
- 古いライブラリ・フレームワークを使用しているシステム

> C#・Java・Python などのマネージド言語はランタイムがメモリを管理するため、古典的なスタックバッファオーバーフローは原則として発生しない。ただし、ネイティブコードとの連携部分（P/Invoke 等）には注意が必要。

---

## 4. 対策1：安全な言語・フレームワークを使用する

メモリ管理を自動で行うマネージド言語を使用することで、バッファオーバーフローのリスクを根本的に排除する。

| 言語 | メモリ管理 | バッファオーバーフローリスク |
| --- | --- | --- |
| C / C++ | 手動 | 高い（直接メモリ操作が可能） |
| C# / Java / Python | 自動（GC） | 低い（ランタイムが境界チェック） |
| Rust | 所有権システム | 低い（コンパイル時に安全性を保証） |

---

## 5. 対策2：入力値の長さを検証する

外部から受け取るデータは、バッファに格納する前に長さを確認する。

```csharp
// ❌ NG：入力値の長さを確認せずそのまま使用する
string name = Request.Form["name"];
ProcessName(name);  // name が異常に長い場合に問題が起きる可能性

// ✅ OK：最大長を定義し、超えた場合は拒否する
const int MaxNameLength = 100;
string name = Request.Form["name"];

if (name.Length > MaxNameLength)
{
    return BadRequest($"名前は{MaxNameLength}文字以内で入力してください。");
}
ProcessName(name);
```

```csharp
// Data Annotations でモデルレベルでのバリデーション
public class UserModel
{
    [Required]
    [StringLength(100, ErrorMessage = "名前は100文字以内で入力してください。")]
    public string Name { get; set; } = string.Empty;

    [Range(0, 150)]
    public int Age { get; set; }
}
```

## 6. 対策3：ライブラリ・フレームワークを最新の状態に保つ

既知のバッファオーバーフロー脆弱性はパッチで修正される。定期的にアップデートを行う。

```bash
# .NET のバージョン確認
dotnet --version

# NuGet パッケージの脆弱性チェック
dotnet list package --vulnerable

# パッケージのアップデート
dotnet add package パッケージ名 --version 最新バージョン
```

## 7. 対策4：ネイティブコードとの連携部分を慎重に扱う

C# で P/Invoke（ネイティブ DLL 呼び出し）を使う場合は、バッファサイズを明示的に管理する。

```csharp
// ❌ NG：バッファサイズを考慮せずネイティブ関数にデータを渡す
[DllImport("native.dll")]
static extern void ProcessData(string input);

// ✅ OK：入力長を事前にチェックし、安全なマーシャリングを使用する
[DllImport("native.dll", CharSet = CharSet.Unicode)]
static extern void ProcessData([MarshalAs(UnmanagedType.LPWStr)] string input, int length);

// 呼び出し前に長さを検証
if (input.Length > MaxBufferSize)
    throw new ArgumentException("入力データが大きすぎます。");

ProcessData(input, input.Length);
```

## 8. OS・サーバーレベルの対策

| 対策 | 内容 |
| --- | --- |
| ASLR（アドレス空間配置のランダム化） | メモリ配置をランダム化し、攻撃コードの実行先を予測困難にする |
| DEP / NX ビット | スタック・ヒープ領域でのコード実行を禁止する |
| スタックカナリア | バッファとリターンアドレスの間に検知値を置き、上書きを検出する |
| 最小権限での実行 | プロセスの実行権限を最小化して被害範囲を限定する |

---

## 9. まとめ：バッファオーバーフロー対策チェックリスト

| 対策 | 内容 |
| --- | --- |
| マネージド言語を使う | C# / Java 等でメモリ管理をランタイムに任せる |
| 入力長を検証する | バッファに格納する前に最大長チェックを行う |
| 定期的にアップデート | ライブラリ・OS のセキュリティパッチを適用する |
| P/Invoke は慎重に | ネイティブコードとの境界でバッファサイズを明示的に管理する |
| 最小権限で実行する | 万一の攻撃成功時の被害範囲を最小化する |
