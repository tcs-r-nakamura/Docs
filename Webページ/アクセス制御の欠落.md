# アクセス制御・認可制御の欠落

---

## 1. アクセス制御・認可制御の欠落とは

ウェブアプリケーションが**「誰が・何にアクセスできるか」を正しく制御していない**脆弱性。URLを直接入力したり、パラメータを改ざんしたりすることで、本来アクセスできないページや他人のデータに到達できてしまう。

### 認証と認可の違い

| 用語 | 意味 |
| --- | --- |
| **認証（Authentication）** | 「あなたは誰か」を確認する（ログイン） |
| **認可（Authorization）** | 「あなたは何をしていいか」を制御する（権限チェック） |

---

## 2. 発生しうる脅威

### 強制ブラウジング（URL直打ち）

- ログインしていないのに認証後のページに直接アクセスできる
- 管理者専用ページに一般ユーザーがアクセスできる

### 他人のデータへのアクセス（IDOR: Insecure Direct Object Reference）

- URL の ID を変えるだけで他ユーザーの情報が閲覧・変更できる

```
// 攻撃例：自分の注文を確認するURL
GET /api/orders/1001    ← 自分の注文

// ID を変えるだけで他人の注文が取れる
GET /api/orders/1002    ← 他人の注文（本来アクセス不可のはず）
```

### 機能レベルのアクセス制御不備

- 一般ユーザーが管理者専用の API を叩ける
- ロールが違うのに同じ画面・機能を使える

---

## 3. 注意が必要なウェブサイトの特徴

| ケース | 例 |
| --- | --- |
| 複数の権限ロールがある | 一般ユーザー・管理者・スタッフなど |
| URL に ID が含まれる | `/orders/123`・`/users/456/edit` |
| 非公開情報を扱う | 個人情報・決済情報・医療情報 |
| ファイルを配信する | 領収書・帳票・添付ファイルのダウンロード |

---

## 4. 対策1：すべてのリクエストで認証・認可チェックを行う

ページ・API の入口だけでなく、**すべてのリクエストに対して**チェックを行う。フロントエンドで非表示にするだけでは不十分。

```csharp
// ASP.NET Core：[Authorize] 属性で認証を要求する
[Authorize]
public class OrderController : Controller
{
    // このコントローラー全体でログインが必須になる
}

// 特定のロールのみ許可する
[Authorize(Roles = "Admin")]
public class AdminController : Controller { }

// ポリシーベースの認可
[Authorize(Policy = "CanEditOrders")]
public IActionResult Edit(int id) { ... }
```

```csharp
// Program.cs：デフォルトで認証を要求する（未指定のエンドポイントも保護）
builder.Services.AddAuthorization(options =>
{
    options.FallbackPolicy = new AuthorizationPolicyBuilder()
        .RequireAuthenticatedUser()
        .Build();
});
```

## 5. 対策2：他人のリソースへのアクセスを防ぐ（IDOR 対策）

リソースの ID だけでなく、**そのリソースが現在のユーザーのものかを必ず確認する**。

```csharp
// ❌ NG：ID の存在だけをチェックし、所有者を確認しない
public async Task<IActionResult> GetOrder(int orderId)
{
    var order = await _db.Orders.FindAsync(orderId);
    if (order == null) return NotFound();
    return Ok(order);  // 他人の注文でも返してしまう
}

// ✅ OK：現在のユーザーの注文かどうかを確認する
public async Task<IActionResult> GetOrder(int orderId)
{
    var userId = int.Parse(User.FindFirstValue(ClaimTypes.NameIdentifier)!);

    var order = await _db.Orders
        .Where(o => o.Id == orderId && o.UserId == userId)  // 所有者チェック
        .FirstOrDefaultAsync();

    if (order == null) return NotFound();  // 存在しない or 他人のものは 404
    return Ok(order);
}
```

## 6. 対策3：機能レベルのアクセス制御をサーバー側で行う

管理者機能はUIで非表示にするだけでなく、**サーバー側でも必ずチェックする**。

```csharp
// ❌ NG：フロントエンドで管理者ボタンを非表示にするだけ
// → URL を直打ちされたら到達できる

// ✅ OK：サーバー側でロールを確認する
[Authorize(Roles = "Admin")]
[HttpDelete("/api/users/{id}")]
public async Task<IActionResult> DeleteUser(int id)
{
    await _userService.DeleteAsync(id);
    return NoContent();
}
```

## 7. 対策4：直接オブジェクト参照を避ける

URL やフォームに内部の ID をそのまま使わず、**間接参照（マッピング）を使う**。

```csharp
// ❌ NG：DB の主キーをそのまま URL に使う
GET /download?fileId=1001  // 1002, 1003 と変えれば他人のファイルが取れる

// ✅ OK：一時的なトークン（UUID等）を発行して間接参照する
var token = Guid.NewGuid().ToString();
_cache.Set(token, new FileReference { FileId = 1001, UserId = currentUserId });
return Ok(new { downloadUrl = $"/download?token={token}" });

// ダウンロード時にトークンを検証
public IActionResult Download(string token)
{
    if (!_cache.TryGetValue(token, out FileReference fileRef))
        return NotFound();

    if (fileRef.UserId != currentUserId)
        return Forbid();

    return File(...);
}
```

## 8. 対策5：ファイル・帳票のダウンロードも認可チェックを行う

ダウンロード URL を知っていれば誰でもアクセスできる設計は危険。

```csharp
// ❌ NG：ファイルパスを URL に含める
GET /files/invoices/invoice_user123.pdf  // パスを推測されると他人の請求書が取れる

// ✅ OK：ダウンロードエンドポイントで認可チェックを行う
[Authorize]
[HttpGet("/api/invoices/{id}/download")]
public async Task<IActionResult> DownloadInvoice(int id)
{
    var invoice = await _db.Invoices.FindAsync(id);
    if (invoice == null) return NotFound();

    // 自分の請求書かチェック
    if (invoice.UserId != CurrentUserId) return Forbid();

    var fileBytes = await _storageService.GetAsync(invoice.FilePath);
    return File(fileBytes, "application/pdf", $"invoice_{id}.pdf");
}
```

---

## 9. まとめ：アクセス制御・認可制御の欠落 対策チェックリスト

| 対策 | 内容 |
| --- | --- |
| 全リクエストで認証チェック | `[Authorize]` をデフォルトで適用し、公開ページのみ `[AllowAnonymous]` |
| 所有者チェック（IDOR対策） | リソースIDだけでなく、現在のユーザーのものかを必ず確認する |
| サーバー側でロール確認 | UIで非表示にするだけでなく、APIにも `[Authorize(Roles="Admin")]` を付ける |
| 間接参照を使う | DB の主キーをそのまま URL に使わず、一時トークンで間接参照する |
| ファイルも認可チェック | ダウンロードエンドポイントでも所有者・権限を確認する |
| デフォルト拒否の原則 | 明示的に許可したもの以外はすべて拒否する設計にする |
