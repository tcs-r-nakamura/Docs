# パス名パラメータの未チェック／ディレクトリ・トラバーサル

---

## 1. ディレクトリ・トラバーサルとは

外部から受け取ったパラメータをそのままファイルパスに使用することで、意図しないディレクトリのファイルにアクセスされる攻撃。`../`（親ディレクトリへの移動）を繰り返すことで、公開を想定していないファイルに到達できてしまう。

```
// 攻撃例：ファイル名パラメータに以下を指定される
GET /download?file=../../../etc/passwd

// 結果として読み込まれるパス
/var/www/html/files/../../../etc/passwd
→ /etc/passwd（OS のユーザー情報が漏えい）
```

---

## 2. 発生しうる脅威

### サーバ内ファイルの閲覧・改ざん・削除

- **重要情報の漏えい**
  - 個人情報・認証情報・APIキーなどが含まれるファイルの不正取得
  - データベースの接続情報（接続文字列）の漏えい
- **設定ファイル・データファイル・プログラムソースコードの改ざん・削除**
  - `appsettings.json`、`.env`、`web.config` などの書き換え
  - アプリケーションロジックの破壊

---

## 3. 注意が必要なウェブサイトの特徴

運営主体やウェブサイトの性質を問わず、**外部からのパラメータにウェブサーバ内のファイル名を直接指定しているウェブアプリケーション**に起こりうる問題。個人情報等の重要情報をウェブサーバ内にファイルとして保存しているサイトは特に注意が必要。

### 該当しやすい実装例

- ウェブページのデザインテンプレートをファイルから読み込む
- 利用者からの入力内容を指定のファイルへ書き込む
- ダウンロード機能でファイル名を URL パラメータで受け取る
- レポート出力でテンプレートファイルを動的に切り替える

---

## 4. 対策1：外部パラメータでファイル名を直接指定する実装を避ける

ファイルの選択をパラメータで直接行わず、**識別子（ID・キー）を受け取り、サーバー側でファイル名に変換する**。

```csharp
// ❌ NG：ファイル名をそのままパラメータで受け取る
string fileName = Request.Query["file"];  // "../../etc/passwd" が渡されうる
var content = File.ReadAllText("/var/files/" + fileName);

// ✅ OK：ID を受け取り、サーバー側でファイル名を決定する
var fileMap = new Dictionary<string, string>
{
    ["template1"] = "report_template_a.html",
    ["template2"] = "report_template_b.html",
};

string templateId = Request.Query["template"];
if (!fileMap.TryGetValue(templateId, out var safeFileName))
{
    return BadRequest("不正なテンプレート指定です。");
}

var content = File.ReadAllText(Path.Combine("/var/templates/", safeFileName));
```

## 5. 対策2：固定ディレクトリを指定し、ファイル名にディレクトリ名が含まれないようにする

```csharp
// ❌ NG：パス全体をパラメータから構築する
string filePath = Path.Combine(baseDir, userInput);  // "../" が含まれると脱出できる

// ✅ OK：ファイル名のみを取り出し、固定ディレクトリと結合する
string baseDir  = @"C:\App\Files\";
string fileName = Path.GetFileName(userInput);  // ディレクトリ部分を除去

// パス結合後に必ず意図したディレクトリ内か確認する
string fullPath = Path.GetFullPath(Path.Combine(baseDir, fileName));

if (!fullPath.StartsWith(baseDir, StringComparison.OrdinalIgnoreCase))
{
    // ディレクトリ外へのアクセス → 拒否
    return Forbid();
}

var content = File.ReadAllText(fullPath);
```

> **ポイント：** `Path.GetFullPath` で正規化してから、ベースディレクトリ内に収まっているかを確認する。

## 6. 対策3：ファイル名のチェックを行う

許可する文字・拡張子をホワイトリストで制限する。

```csharp
// ファイル名に使用できる文字をホワイトリストで制限
bool IsValidFileName(string fileName)
{
    // ディレクトリ区切り文字・特殊文字を含む場合は拒否
    if (fileName.IndexOfAny(Path.GetInvalidFileNameChars()) >= 0)
        return false;

    // ".." を含む場合は拒否
    if (fileName.Contains(".."))
        return false;

    // 許可する拡張子のみ受け付ける
    var allowedExtensions = new HashSet<string> { ".pdf", ".csv", ".xlsx" };
    string ext = Path.GetExtension(fileName).ToLowerInvariant();
    return allowedExtensions.Contains(ext);
}

// 使い方
string fileName = Request.Query["file"];
if (!IsValidFileName(fileName))
{
    return BadRequest("許可されていないファイル名です。");
}
```

## 7. 対策4：ウェブサーバ内のファイルへのアクセス権限を正しく管理する

万一パス操作に成功しても、被害を最小限に抑える多層防御。

| 場所 | 設定内容 |
| --- | --- |
| 公開ディレクトリ | ウェブサーバーが読み取れるファイルのみ配置 |
| 設定ファイル（`appsettings.json` 等） | ウェブサーバーの実行ユーザーからアクセスできないディレクトリに配置 |
| OS のファイルパーミッション | 書き込み不要なファイルは読み取り専用に設定 |
| ルートディレクトリ | ウェブアプリの実行ユーザーに `/`（ルート）へのアクセス権を与えない |

---

## 8. まとめ：ディレクトリ・トラバーサル対策チェックリスト

| 対策 | 内容 |
| --- | --- |
| 直接指定を避ける | ファイル名ではなく ID を受け取り、サーバー側でマッピングする |
| 固定ディレクトリ内に限定 | `Path.GetFullPath` で正規化後、ベースディレクトリ内か確認する |
| ファイル名を検証 | `..` や無効文字を含む場合は拒否。拡張子はホワイトリストで制限する |
| アクセス権限を最小化 | 設定ファイル等は公開ディレクトリ外に置き、OS レベルで権限を制限する |
