# クリックジャッキング

---

## 1. クリックジャッキングとは

正規サイトを透明な `<iframe>` で重ね、利用者が**意図しないボタン・リンクをクリックさせる**攻撃。利用者には罠サイトのコンテンツが見えているが、実際にはその裏で正規サイトを操作している。

```
攻撃の仕組み：

罠サイト（利用者に見えているもの）
┌─────────────────────────────┐
│  プレゼントに応募する！       │
│  ┌───────────┐               │
│  │ クリック！ │               │  ← 利用者がクリック
│  └───────────┘               │
└─────────────────────────────┘

実際の重なり（透明な iframe）
┌─────────────────────────────┐
│  銀行サイト（透明・不可視）   │
│  ┌───────────┐               │
│  │ 送金する  │               │  ← 実際に押されているボタン
│  └───────────┘               │
└─────────────────────────────┘
```

---

## 2. 発生しうる脅威

- **不正な送金・決済**：銀行・ショッピングサイトの操作ボタンを押させる
- **意図しないアカウント操作**：退会・設定変更・投稿ボタンを押させる
- **SNS の不正操作**：「いいね」・「フォロー」・「シェア」を押させる
- **管理者権限の悪用**：管理者画面の危険な操作を実行させる

---

## 3. 注意が必要なウェブサイトの特徴

ログイン機能・決済機能・重要な操作ボタンを持つウェブサイト全般。

| カテゴリ | 例 |
| --- | --- |
| 金銭処理が発生するサイト | ネットバンキング・ショッピングサイト |
| アカウント操作が可能なサイト | SNS・メール・会員サービス |
| 管理者機能を持つサイト | CMS・管理者画面 |

---

## 4. 対策1：X-Frame-Options ヘッダを設定する（主要対策）

レスポンスヘッダで自サイトが `<iframe>` に埋め込まれることを禁止する。

```csharp
// ASP.NET Core：Middleware でヘッダを追加する
app.Use(async (context, next) =>
{
    // DENY：いかなるサイトの iframe にも表示しない（最も厳格）
    context.Response.Headers.Add("X-Frame-Options", "DENY");

    // または SAMEORIGIN：同じオリジンの iframe のみ許可する
    // context.Response.Headers.Add("X-Frame-Options", "SAMEORIGIN");

    await next();
});
```

| 値 | 動作 |
| --- | --- |
| `DENY` | すべての iframe 埋め込みを禁止（推奨） |
| `SAMEORIGIN` | 同じオリジンからの iframe のみ許可 |
| `ALLOW-FROM uri` | 指定した URI からの iframe のみ許可（非推奨・廃止予定） |

---

## 5. 対策2：Content-Security-Policy の frame-ancestors を設定する

`X-Frame-Options` より柔軟に制御できる新しい方法。複数オリジンの指定も可能。

```csharp
// ASP.NET Core：CSP ヘッダで iframe 埋め込みを制御する
app.Use(async (context, next) =>
{
    // iframe への埋め込みを完全に禁止
    context.Response.Headers.Add(
        "Content-Security-Policy",
        "frame-ancestors 'none'");

    // または同一オリジンのみ許可
    // "frame-ancestors 'self'"

    await next();
});
```

| 設定値 | 動作 |
| --- | --- |
| `frame-ancestors 'none'` | すべての iframe 埋め込みを禁止 |
| `frame-ancestors 'self'` | 同一オリジンのみ許可 |
| `frame-ancestors https://trusted.com` | 指定ドメインのみ許可 |

---

## 6. 対策3：重要操作に再確認を挟む（多層防御）

クリックジャッキングが成立しても被害を最小化するため、送金・退会などの重要操作には**確認ページ・パスワード再入力**を設ける。

```csharp
// 重要操作の前に確認ページを挟む
[HttpPost]
[ValidateAntiForgeryToken]
public IActionResult Transfer(TransferModel model)
{
    // 確認画面に遷移（1クリックで完了させない）
    TempData["TransferModel"] = JsonSerializer.Serialize(model);
    return RedirectToAction("Confirm");
}

[HttpPost]
[ValidateAntiForgeryToken]
public async Task<IActionResult> ConfirmTransfer(string password)
{
    // パスワード再確認
    if (!await _authService.VerifyPasswordAsync(User, password))
        return Unauthorized();

    var model = JsonSerializer.Deserialize<TransferModel>(TempData["TransferModel"].ToString());
    await _bankService.TransferAsync(model);
    return RedirectToAction("Complete");
}
```

---

## 7. まとめ：クリックジャッキング対策チェックリスト

| 対策 | 内容 |
| --- | --- |
| `X-Frame-Options: DENY` | すべてのページで iframe 埋め込みを禁止する |
| `CSP: frame-ancestors 'none'` | より柔軟に iframe 制御できる新しい方法 |
| 重要操作に確認ページを挟む | 1クリックで完了する設計にしない |
| CSRF トークンと組み合わせる | クリックジャッキング + CSRF の複合攻撃にも対応する |
